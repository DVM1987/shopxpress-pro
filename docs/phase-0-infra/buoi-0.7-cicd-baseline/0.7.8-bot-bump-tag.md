## TL;DR

GitHub Actions workflow #2 (`bump-tag.yml`) tự động sửa `image.tag` trong repo manifest (`shopxpress-pro-deploy`) mỗi khi `build-push.yml` push image mới lên ECR. ArgoCD poll repo manifest → sync pod tag mới. Hoàn tất chuỗi GitOps pull-based: `git push code → pod live` không touch tay.

## Tại sao tách 2 workflow (không gộp 1 file)

| Tách | Gộp |
|---|---|
| Trigger độc lập (push code vs deploy) | 1 trigger duy nhất |
| Bot fail KHÔNG block build | 1 fail block toàn bộ |
| Permission tách: build-push cần OIDC AWS, bump-tag cần SSH key | Cộng dồn permission, vi phạm least privilege |
| Re-run riêng (build OK, bump fail → chỉ rerun bump) | Phải rerun cả build |

→ Industry pattern: **CI workflow ≠ CD workflow**, kết nối qua `workflow_run`.


## Pattern senior chốt

### 1. SSH deploy key thay vì PAT
- Deploy key gắn **repo cụ thể**, không gắn user → least privilege.
- User nghỉ team → key vẫn sống, bus factor = 0 dependency.
- Lộ key chỉ ảnh hưởng 1 repo (PAT lộ → mất tất cả repo của user).

### 2. workflow_run trigger
- `types: [completed]` bao gồm cả `success` lẫn `failure` → job phải tự gate `if: workflow_run.conclusion == 'success'`.
- `branches: [main]` chỉ react khi `build-push` chạy trên main (PR branch không trigger — vì PR chưa push image lên ECR).
- Workflow definition load từ DEFAULT branch tip, KHÔNG phải SHA gốc → fix bug phải merge vào main mới có hiệu lực; rerun chỉ chạy lại với code cũ ở SHA gốc.

### 3. paths-filter re-run
- `workflow_run` chạy ở context riêng, không tự biết file nào đổi → phải re-run paths-filter.
- Default mode (không pass `ref/base`) diff `HEAD vs HEAD~1` → cần checkout `fetch-depth: 2`.

### 4. max-parallel: 1 + retry rebase
- 3 service có thể trigger song song → 3 git push race → non-fast-forward.
- `max-parallel: 1` queue tuần tự + retry 3x với `git pull --rebase` defensive.

### 5. Idempotent skip
- Bot tự skip commit nếu `current_tag == new_tag` → không tạo commit rỗng → không trigger ArgoCD sync vô ích.

### 6. Scope env=dev only
- Stg/prd auto-promote là **anti-pattern** — mất kiểm soát compliance/audit.
- Stg/prd manual gate qua PR review ở repo deploy (Buổi 14 canary sẽ làm).

## 2 pitfall đã fix

### Pitfall 1: paths-filter `base: <SHA>~1` invalid refspec
```
fatal: invalid refspec '5a22e2891058c4f89079e9e08d695248cea56fc4~1'
```
- `dorny/paths-filter@v3` truyền `base` thẳng vào `git fetch` mà fetch không parse `<SHA>~1` syntax.
- Fix: bỏ `ref` + `base`, dùng default mode (HEAD vs HEAD~1) — checkout `fetch-depth: 2` đủ context.

### Pitfall 2: workflow_run rerun không pick code mới
- Sau khi merge fix vào main, rerun failed run cũ → vẫn dùng workflow definition của SHA gốc (chứa bug).
- Lý do: rerun = chạy lại cùng commit context, KHÔNG load workflow mới.
- Workaround: tạo commit mới (dummy 1 file service code) → `build-push` trigger lại → workflow_run mới sẽ dùng workflow definition ở main tip.

## Gộp fix Sub-comp 0.7.7.1 vào 0.7.8

`build-push.yml` `IMAGE_PLATFORM: linux/arm64` → `linux/amd64`. EKS MNG nodes là `amd64`, image arm64 → ImagePullBackOff `no match for platform`. Gộp vào commit 0.7.8 vì cùng đụng workflow file.

## Interview takeaways (Senior 5+)

1. **GitOps pull vs push**: ArgoCD KHÔNG bao giờ touch CI. Bot là cây cầu duy nhất app↔manifest, tách audit log.
2. **SSH deploy key > PAT cho cross-repo write**: bus factor + blast radius.
3. **Tách CI ≠ CD**: build và deploy 2 workflow riêng, kết nối `workflow_run`.
4. **Auto-bump CHỈ dev**: prod phải PR thủ công (compliance + change advisory).
5. **Idempotent bot**: skip commit khi tag không đổi → tránh trigger sync rỗng.
6. **paths-filter trong workflow_run** phải re-run vì context mới không có change info.
7. **Concurrency tuần tự cho job push git**: tránh race non-fast-forward.
