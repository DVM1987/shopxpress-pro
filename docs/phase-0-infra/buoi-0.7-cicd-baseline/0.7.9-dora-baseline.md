# Sub-comp 0.7.9 — DORA Baseline (Lead Time + Deploy Frequency)

**Ngày**: 2026-05-10
**Scope**: Đo 2 trong 4 metric DORA từ pipeline CI/CD đã chạy thực tế ở Buổi 0.7. CFR (Change Failure Rate) + MTTR (Mean Time to Restore) sẽ làm ở Phase 2 Observability khi đã có Prometheus + Alerting.
**Method**: Lab-first — kéo timestamp thật từ `gh run`, `git log` ở 2 repo `shopxpress-pro-app` (CI) + `shopxpress-pro-deploy` (CD bot commit), tính delta giữa các stage. KHÔNG dùng tool chuyên dụng (Sleuth, LinearB) — manual để hiểu cơ chế.

## 1. Lead Time for Changes — Mẫu commit `4a2cfe4 → 179d0aa`

**Định nghĩa DORA** (Accelerate, Forsgren et al.): khoảng thời gian từ lúc code được commit đến khi code chạy thật trong production.

**Trong Lab A++ định nghĩa cụ thể**:
- **Start**: commit timestamp của developer ở branch dev (`git commit` author time).
- **End**: pod K8s với image tag mới ở trạng thái `Ready` phục vụ traffic ở namespace `dev` (proxy production vì stg/prd parked replicas=0).

### 1.1 Timeline e2e đo được

Dữ liệu nguồn:
- `git log --pretty='%H %aI'` ở repo app cho commit dev.
- `gh run view 25623592092` cho build-push.
- `gh run view 25623618163` cho bump-tag.
- `git log` ở repo deploy cho bot commit.
- ArgoCD sync time đo lại bằng `kubectl get events -n argocd` (memory ghi ~12s sau hard refresh, fallback poll mặc định 3p).

| # | Event | UTC | Δ from commit |
|---|---|---|---|
| 1 | Commit `4a2cfe4` author (dev branch `test/bump-tag-e2e`) | `2026-05-10T08:02:38Z` | 0 |
| 2 | PR #6 squash merge → main `179d0aa` (build-push run created) | `08:04:11Z` | +1m33s |
| 3 | build-push job `gateway` started | `08:04:22Z` | +1m44s |
| 4 | build-push job `gateway` completed (image `sha-179d0aa` lên ECR) | `08:05:28Z` | +2m50s |
| 5 | Bot commit `6324dc4` ở repo deploy (`bot: bump gateway dev → sha-179d0aa`) | `08:05:44Z` | +3m06s |
| 6 | bump-tag run completed | `08:05:49Z` | +3m11s |
| 7a | ArgoCD detect commit + sync (auto poll mặc định 3p worst-case) | `~08:08:44Z` | +6m06s |
| 7b | Pod `gateway-dev-*` `2/2 Running` `sha-179d0aa` (rolling ~30-60s) | `~08:09:14Z` | **~6m36s** |

**Lead Time observed** ≈ **6m36s** với ArgoCD poll mặc định.

### 1.2 Stage breakdown (tách 4 phase rõ ràng)

| Stage | Owner | Time | % total | Optimization |
|---|---|---|---|---|
| A. Code review + merge (PR open → squash to main) | Human | 1m33s | 23% | Lab 1 dev nên ngắn — production thực tế ~hours-days, đây là "non-deterministic" component |
| B. CI build + scan + push ECR | GHA | 1m17s | 19% | Cache buildx GHA scope per service, Trivy scan parallel — đã optimal cho 1 service nhỏ |
| C. CD bot bump tag (workflow_run → git push deploy) | GHA bot | 21s | 5% | SSH deploy key ngay khi build-push success, race-safe với rebase retry 3x |
| D. ArgoCD sync + rolling update | ArgoCD | ~3m25s | 53% | **Bottleneck**. Webhook thay vì poll giảm xuống <30s. Sub-comp Buổi 14 sẽ làm |

**Insight**: nửa thời gian Lead Time trong baseline này là chờ ArgoCD poll. Một webhook GitHub → ArgoCD `Application` `refresh` sẽ đưa Lead Time xuống ~3m30s (chỉ stage A+B+C+rolling). Đó là lý do production GitOps nghiêm túc luôn dùng webhook.

### 1.3 Phân loại DORA (theo State of DevOps Report)

| Tier | Lead Time |
|---|---|
| **Elite** | < 1 hour |
| High | 1 day - 1 week |
| Medium | 1 week - 1 month |
| Low | > 1 month |

→ **Elite** (~6m). Lưu ý: sample size = 1 commit, lab service nhỏ chỉ có Go vet + race test, không có integration test phức tạp. Production thực tế Lead Time thường 30m-2h tier Elite vì test suite + manual approval gate.

## 3. Cách đo lặp lại (script hoá tương lai)

```bash
# Lead Time của 1 commit
COMMIT_SHA=4a2cfe4
APP_REPO=DVM1987/shopxpress-pro-app
DEPLOY_REPO=DVM1987/shopxpress-pro-deploy

# Stage A: commit timestamp
COMMIT_TS=$(gh api "repos/$APP_REPO/commits/$COMMIT_SHA" --jq '.commit.author.date')

# Stage B: build-push run for merge SHA (cần map dev SHA → main SHA bằng PR)
PR_NUM=$(gh pr list --repo "$APP_REPO" --search "$COMMIT_SHA" --state merged --json number --jq '.[0].number')
MERGE_SHA=$(gh pr view "$PR_NUM" --repo "$APP_REPO" --json mergeCommit --jq '.mergeCommit.oid')
BUILD_DONE=$(gh run list --workflow=build-push.yml --commit "$MERGE_SHA" --json updatedAt --jq '.[0].updatedAt')

# Stage C: bot commit ở deploy repo (search by image tag prefix sha-<7chars>)
TAG=sha-$(echo $MERGE_SHA | cut -c1-7)
BOT_COMMIT_TS=$(gh api "repos/$DEPLOY_REPO/commits?per_page=20" \
  --jq ".[] | select(.commit.message | contains(\"$TAG\")) | .commit.author.date" | head -1)

# Stage D: pod ready (cần Prometheus query hoặc kubectl events archive)
# Phase 2 Observability sẽ dùng kube-state-metrics:
#   kube_pod_status_ready{namespace="dev",pod=~"gateway-.*"} == 1
# kết hợp với image:tag label.
```

Sub-comp Buổi 10 (Phase 2 Observability) sẽ wire metric này tự động vào Grafana dashboard "DORA Lead Time".

## 4. Limitations baseline 0.7.9

1. **Sample size n=1** cho Lead Time, n=3 cho Deploy Frequency → variance không xác định, KHÔNG đại diện. Cần ≥ 1 tuần production cadence.
2. **Stage A (PR review) không deterministic** — phụ thuộc cycle developer/reviewer. Lab 1 người nên ~1m33s không representative.
3. **CFR + MTTR chưa đo** — cần Prometheus alert + incident timeline. Phase 2.
4. **Promotion path stg/prd manual** — Deploy Frequency thực sự "to prod" sẽ < 1/day cho đến khi Buổi 14 canary tự động hoá.
5. **ArgoCD poll vs webhook** — baseline dùng poll mặc định 3p. Webhook chưa setup → Lead Time stage D inflate gấp 3-4 lần realistic.

## 6. Câu phỏng vấn Senior 5+ rút từ 0.7.9

1. **"Lead Time của bạn đo từ đâu đến đâu, và stage nào là bottleneck?"** — Trả lời: từ commit author timestamp ở dev branch tới pod Ready ở namespace prod. Bottleneck của baseline lab này là Stage D (ArgoCD poll 3p) chiếm 53%, fix bằng webhook giảm xuống <30s.

2. **"Vì sao đo Deploy Frequency trên build success thay vì merge to main?"** — Vì merge to main không guarantee image push thành công (Trivy gate có thể fail HIGH/CRITICAL). Build success là proxy đáng tin hơn. Strictest definition là pod Ready với image mới — đó mới là "deploy thực sự".

3. **"DORA classification của bạn dựa trên sample size bao nhiêu, và tại sao quan trọng?"** — n=1 cho Lead Time, n=3 cho Deploy Frequency → không đại diện. State of DevOps Report dùng survey ≥1000 dev rolling 6 tháng. Production thực tế cần rolling window 7-30 ngày, đo theo team chứ không phải repo.

4. **"Khác biệt giữa Lead Time for Changes (DORA) và Cycle Time (Lean) là gì?"** — DORA Lead Time bắt đầu từ commit, không tính requirement → design → review → coding. Cycle Time Lean tính từ "work started" (ticket "in progress"). DORA dễ đo automation, Cycle Time đòi hỏi tooling Jira/Linear.

5. **"Tại sao nửa thời gian Lead Time của bạn nằm ở ArgoCD sync, và bạn fix thế nào trong production?"** — Vì poll mặc định 3 phút. Fix bằng webhook GitHub → ArgoCD `argocd-server` `/api/webhook` endpoint. Auth bằng shared secret. Sub-comp Buổi 14 sẽ làm khi tới canary cần precise Lead Time.

6. **"Nếu CFO hỏi 'DORA tier mình ở đâu', bạn trả lời sao mà không bị phản đối?"** — Trả lời theo strict definition (deploy to production with verified Ready pod) — High tier 1/day baseline. KHÔNG inflate bằng cách đếm CI builds. Show breakdown 4 stage để CFO thấy đâu là eng cost vs human cost.
